# 修订模式：三级存量修订

> **模块属性**：独立于主流水线的修订功能模块，拥有独立触发入口。不改动原有四阶段新建创作基线能力，与主流水线双向可切换，修订完成后自动同步所有底层索引，无需重启会话。

---

## 总览

```
修订模式三级结构（强制自上而下执行，禁止逆向）

┌─────────────────────────────────────────────┐
│ 模式一：基础设定扫描更新（顶层修订）           │
│ 范围：00-世界观.md + 01-人物卡/              │
├─────────────────────────────────────────────┤
│ 模式二：大纲体系扫描更新（中层修订）           │
│ 范围：02-大纲/ 六份文件                       │
├─────────────────────────────────────────────┤
│ 模式三：正文双分支修订（底层修订）             │
│ 分支1：剧情结构性修改                         │
│ 分支2：文本质感润色打磨                       │
│ 范围：已产出章节文件                          │
└─────────────────────────────────────────────┘
```

---

## 准入规则

### 禁用时序
- Phase 0 初始化阶段：禁用
- Phase 1 用户提问阶段：禁用

### 解锁时序
- Phase 2 设定文件全部生成完毕后：永久解锁
- Phase 3 创作过程中：可用（独立触发，不暂停主创作线程）
- Phase 4 校验过程中：可用

### 触发词示例
- 「修改世界观」「调整世界设定」
- 「修改主角人设」「调整人物卡」
- 「修改大纲」「调整卷纲」「改写时间线」
- 「改写第X章」「重写第X章」
- 「润色第X章」「去AI味」「优化文风」
- 「修订设定」「检查设定一致性」

### 内部时序约束
强制自上而下执行：基础设定修订 → 大纲修订 → 正文修订
禁止逆向修改（正文修订不得反向修改大纲，大纲修订不得反向修改世界观）

## 模式一：基础设定扫描更新模式（顶层修订）

### 范围
- `00-世界观.md`
- `01-人物卡/主角.md`
- `01-人物卡/重要角色.md`
- `01-人物卡/次要角色.md`
- `01-人物卡/重要反派.md`
- `01-人物卡/次要反派.md`

### 步骤 1：前置自动扫描

Agent 遍历扫描 `00-世界观.md`、`01-人物卡/` 全量档案，完成三项合规扫描：

**① 设定内部逻辑冲突**
- 世界观力量体系规则是否自洽
- 角色固有属性是否与世界观匹配
- 势力关系是否矛盾（盟友变敌人无铺垫等）

**② 主角能力提升规则匹配度**
- 主角能力提升规则与人设是否匹配（如：性格懒惰但修为飞速提升）
- 主角能力提升规则与世界观底层规则是否兼容

**③ 下游一致性偏差**
- 设定与 `02-大纲/` 是否存在矛盾
- 设定与已生成正文是否存在冲突
- 设定与 `topLevelConstraints.coreConflict` 是否偏离

**输出**：《基础设定差异简报》，标注冲突位置与影响范围

### 步骤 2：用户交互与改写

1. 向用户同步差异简报
2. 开放式问询用户修改意图，区分四类需求：
   - **局部微调**：细节补充/措辞优化
   - **细则增补**：新增设定细节（不矛盾）
   - **冗余删减**：删除重复/过时内容
   - **逻辑修正**：修复内部矛盾
3. 执行改写

### 步骤 3：改写边界

**允许**：局部微调、细则增补、冗余删减、逻辑修正

**禁止**：
- 颠覆性推翻世界观底层框架（如删除整个修炼体系）
- 颠覆性修改角色核心人设（如将正义主角改为反派）

**升级处理**：用户提出颠覆性修改需求时，提示：
```
此修改将颠覆世界观底层框架/角色核心人设，存量强行改写会导致后续剧情全面崩坏。
建议：重启 Phase 1 新建项目，从源头重新规划。
是否确认继续（风险自负）/ 放弃此修改 / 重启新建？
```

### 步骤 4：下游联动

设定修改完成后，自动执行：
1. 标记 `02-大纲/` 中受影响的冲突节点
2. 标记受影响的已生成正文章节
3. 输出下游影响清单，提示用户：
   - 如有大纲冲突 → 建议后续执行模式二（大纲修订）
   - 如有正文冲突 → 建议后续执行模式三（正文修订）

## 模式二：大纲体系扫描更新模式（中层修订）

### 范围
- `02-大纲/卷纲.md`
- `02-大纲/时间线.md`
- `02-大纲/伏笔布局.md`
- `02-大纲/进阶条件.md`
- `02-大纲/成长体系.md`
- `02-大纲/境界功法.md`

### 步骤 1：前置自动扫描

全量扫描六份大纲文件，双向比对：
- 当前最新版 `00-世界观.md` + `01-人物卡/`
- `03-写作计划.json` 中的 `topLevelConstraints`（核心矛盾、能力路径）
- 伏笔追踪表

识别五类违规项：

| 违规类型 | 检查内容 |
|---------|---------|
| 世界观违规 | 大纲是否违背当前最新版世界观规则 |
| 能力路径偏离 | 角色成长路径是否偏离 `topLevelConstraints.abilityPath` |
| 人设冲突 | 大纲事件/行为是否与人物卡核心设定冲突 |
| 伏笔时间线错位 | 伏笔排布时间节点是否与时间线自洽 |
| 支线消解主线 | 支线剧情是否消解 `topLevelConstraints.coreConflict` |

**输出**：大纲违规清单（五类违规项逐条列出）

### 步骤 2：用户交互与改写

1. 同步大纲违规清单
2. 问询用户修改需求，支持：
   - 章节增删
   - 分卷合并/拆分
   - 时间线偏移
   - 伏笔增减
   - 双轨成长节点调整
3. 执行改写

### 步骤 3：刚性约束

**必须遵守**：
- 所有大纲修改必须贴合当前最新版世界观、人物卡
- 禁止私自新增势力、功法、角色权限（未在世界观中定义）
- 不允许绕过顶层设定隐性改参

### 步骤 4：下游联动

修订完成后自动同步：
1. 更新 `03-写作计划.json`（章节列表/标题/总数）
2. 更新全局伏笔追踪表（`02-大纲/伏笔布局.md`）
3. 刷新 L1 阶段总结索引（如受影响阶段已完成）
4. 如有已生成正文受影响 → 标记章节，提示执行模式三

## 模式三：正文双分支更新修订模式（底层修订）

### 范围
已产出章节文件（`第XX章-*.md`）

### 分支互斥规则
两个分支互斥、不可并行执行。Agent 根据用户输入语义自动无感分支：
- 涉及剧情/人设/伏笔/时间线改动 → 分支1（剧情结构性修改）
- 仅涉及文字表达优化 → 分支2（文本质感润色打磨）

### 共用前置：三层记忆合规校验

执行任何分支前，必须加载：
1. 最新 L1 阶段总结（`memory/phase-{N}-summary.md`）
2. 相关 L2 章节记忆（`memory/ch-{XXX}-memory.md`）
3. 目标章节 L3 全文（原始章节文件）

复用 Phase 4 五维校验标准（字数/伏笔/时间线/人设/文风）。

---

### 分支 1：剧情结构性修改

**适用场景**：改动章节事件因果、人物临场行为、对话内核、伏笔埋设/回收、支线动线

#### 合规约束（逐项核验）

1. 对齐当前最新版 `00-世界观.md` 底层规则
2. 对齐 `02-大纲/卷纲.md` 中该章的单章大纲规划
3. 对齐 `01-人物卡/` 中相关角色的核心设定
4. 对齐 `topLevelConstraints.abilityPath`（主角能力提升路径）
5. 对齐 `topLevelConstraints.coreConflict`（主线核心矛盾）
6. 跨章节修改需校验 L1 阶段总结连贯性

#### 禁止项
- 私自变更主角变强逻辑（绕过能力路径约束）
- 弱化主线核心矛盾（将核心冲突降级为支线级别）

#### 修改后同步
1. 更新对应 L2 章节记忆文件
2. 更新 `02-大纲/伏笔布局.md` 中的伏笔台账
3. 如修改影响后续章节 → 标记受影响章节

#### Phase 3 中途创作适配

**场景 A：在 Phase 3 创作断点处修改**
- 已生成章节：仅同步更新 L1/L2 记忆索引、伏笔台账，**原文永久锁定不改动**
- 未生成后续章节：强制套用修改后全套约束逻辑（世界观/人设/能力路径/核心矛盾/大纲）
- 草稿章节（status: "draft"）：直接回滚重写

**场景 B：在 Phase 2 创作启动前修改**
- 后续全量章节统一套用修改后约束

---

### 分支 2：文本质感润色打磨

**适用场景**：零剧情、零人设、零伏笔、零时间线改动，仅优化文本表层表达

#### 强制复用基线去AI味规则

逐项执行，缺一不可：
1. 删除高频AI废词：「此刻、见状、不由得、随即、不禁、仿佛、宛如、油然而生、心潮澎湃」
2. 打散对称式四字成语堆砌
3. 优化「的」字冗余密度（单句不超过2个「的」）
4. 丰富长短句交替
5. 补充轻量化场景口语化表达
6. 优化节奏感（段落长短变化，网文阅读适配）
7. 细节具象化（用具体细节替代笼统描述）

#### 文风锁定
全程锁定全文初始文风参数，杜绝文风割裂。修订后文本必须与原文风格一致，不得引入新的文风特征。

#### 润色后校验
- 字数检查：润色后字数不得低于原文（允许 ±5% 浮动）
- AI废词复查：确认废词已清除
- 文风一致性：修订后段落与未修订段落风格统一

## 跨模式时序与算力适配规则

### 时序互斥规则

| 时序 | 修订模式状态 |
|------|------------|
| Phase 0 | 禁用 |
| Phase 1 | 禁用 |
| Phase 2 设定固化后 | 永久解锁 |
| Phase 3 创作中 | 可用（独立触发，不暂停主创作线程） |
| Phase 4 校验中 | 可用 |

### 修订内部时序
强制自上而下：基础设定修订 → 大纲修订 → 正文修订
禁止逆向修改，避免层级逻辑崩坏。

### 断点接续流程

```
修订触发
    ↓
自动留存快照（memory/snapshot.json）
    ↓
执行修订（三级模式对应流程）
    ↓
修订完成，同步底层索引
    ↓
读取快照，恢复创作游标
    ↓
跳过已完成章节，复用原有算力模式继续创作
```

### 并行修订隔离

调用三级修订时：
1. 独立读写修订副本（不直接覆盖原文件）
2. 校验无逻辑冲突后批量原子替换
3. 防止创作读写冲突

### 分层内容差异化处理

| 章节状态 | 处理方式 |
|---------|---------|
| 已完成（completed） | 仅同步更新 L1/L2 记忆索引、伏笔台账，原文永久锁定不改动 |
| 待创作（pending） | 全量套用修订后的世界观、人设、能力路径、主线矛盾、大纲约束 |
| 草稿（draft） | 直接回滚重写 |

### 游标复位接续

修订完成后：
1. 自动读取 `memory/snapshot.json` 中的 `lastCompletedChapter`
2. 跳过全部已完成章节
3. 复用原有算力模式（serial/subagent-parallel/agent-teams）直接继续创作
4. 会话无重启、上下文无断层

## 统一异常兜底机制

### 1. 记忆加载异常
- **触发**：L1/L2 记忆文件损坏或缺失
- **处理**：自动降级兜底读取 L3 上章原文，自动修复损坏记忆分片
- **用户通知**：无（静默修复）

### 2. 超限修改需求
- **触发**：用户提出脱离顶层设定的修改（如在功法体系中加入科技元素）
- **处理**：不直接拒绝，输出合规替代方案，标注风险点位
- **示例**：
  ```
  您要求在功法体系中加入科技元素，这将偏离原始设定的「功法修炼」能力路径。
  替代方案：在世界观中增设「炼器」分支，以功法驱动法器制造，兼容科技感但不脱离修炼体系。
  风险点：需同步更新成长体系、境界功法、相关人物卡。
  是否采用替代方案 / 坚持原修改（风险自负） / 放弃？
  ```

### 3. 修订重写超限
- **触发**：单章节修订超过 3 轮仍未通过校验
- **处理**：终止自动修复，输出异常报告交由用户决策
- **报告内容**：未通过项清单、3轮修改记录、建议处理方式

### 4. 会话中断
- **触发**：修订过程中会话中断
- **处理**：触发 Phase 0 中断检测逻辑，自动恢复原有记忆与设定文件
- **续改**：下次触发时，Phase 0 检测到修订中断状态，询问用户是否继续修订
