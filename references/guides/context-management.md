# 上下文管理机制

> **目标**：在保证章节衔接自然、剧情连贯的前提下，最大限度减少Token消耗。核心思路：用结构化记忆替代全文回读，用阶段压缩替代累积膨胀。

---

## 一、三层记忆架构

```
全书上下文 = 阶段总结（粗粒度） + 近期记忆（细粒度） + 上章全文（衔接用）
```

| 层级 | 文件 | 内容 | Token占用 | 加载时机 |
|------|------|------|----------|---------|
| L1 阶段总结 | `memory/phase-XX-summary.md` | 每10章压缩一次 | ~300-500字 | 始终加载最新1份 |
| L2 章节记忆 | `memory/ch-XXX-memory.md` | 每章结束后生成 | ~150-300字/章 | 加载最近2份 |
| L3 上章全文 | `第XXX章-xxx.md` | 原始章节文本 | ~3000-5000字 | 仅加载上1章 |

**Token对比**：
- 旧方案（读全部章节）：30章 × 4000字 = 120,000字
- 新方案（30章时）：1份阶段总结(500) + 2份章节记忆(600) + 1章全文(4000) = ~5,100字
- **节省约 95% Token**

## 二、章节记忆文件（每章写完后生成）

### 文件路径

`memory/ch-{XXX}-memory.md`

### 结构模板

```markdown
# 第{XXX}章记忆摘要

## 场景收束
- 位置：[本章结束时的场景地点]
- 时间：[具体时间点/季节]
- 氛围：[情绪基调，如紧张/温馨/压抑]

## 人物状态变化
- {角色A}：[情绪/伤势/实力/关系变化]
- {角色B}：[同上]
（仅列出本章有状态变化的角色）

## 剧情推进
- 核心事件：[一句话概括本章主要事件]
- 新信息：[本章揭示的新情报/真相]
- 关系变动：[角色关系的增减/质变]

## 悬念状态
- 已关闭：[本章回收的悬念]
- 新开启：[本章新设的钩子]
- 持续中：[仍未解决的悬念列表]

## 伏笔记录
- 埋设：[本章新埋的伏笔，含预计回收章节]
- 回收：[本章回收的伏笔]
- 推进：[本章推进了哪些伏笔的铺垫]

## 衔接锚点（供下一章使用）
- 结尾场景：[最后一幕的具体场景描述]
- 结尾情绪：[读者此刻的情绪状态]
- 结尾悬念：[最后一个钩子/留白]
- 未完成动作：[正在发生但未结束的事]
```

### 生成规则

1. 在每章正文完成后、更新写作计划之前生成
2. 严格控制在 150-300 字以内
3. 只记录对后续章节有影响的信息，不记录已完结的细节
4. 「衔接锚点」是最重要的部分，必须精准描述章节结尾状态

## 三、阶段总结文件（每10章压缩一次）

### 触发条件

当 `completed` 章节数为 10 的倍数时，自动触发生成/更新。

### 文件路径

`memory/phase-{N}-summary.md`（N = 当前第几个10章，从1开始）

### 生成流程

1. 读取本阶段所有章节记忆文件（ch-{N*10-9} 到 ch-{N*10}）
2. 压缩为一份阶段总结（300-500字）
3. 生成完成后，本阶段的章节记忆文件**不删除**（保留供精确回溯）

### 结构模板

```markdown
# 阶段总结：第{X}-{Y}章

## 剧情主线推进
[本阶段主线从什么状态推进到什么状态，2-3句话]

## 人物弧光进展
- {主角}：[本阶段成长/变化轨迹]
- {关键角色A}：[本阶段关键变化]
- {关键角色B}：[本阶段关键变化]
（仅列出本阶段有显著弧光推进的角色）

## 伏笔总览
- 本阶段新埋：[数量] 条
- 本阶段回收：[数量] 条
- 当前活跃伏笔：[列出仍未回收的关键伏笔]

## 悬念线状态
- {悬念线A}：[当前状态]
- {悬念线B}：[当前状态]

## 世界状态快照
[当前世界的整体状态，势力格局变化等]

## 阶段结尾锚点
- 最新场景：[第Y章结束时的场景]
- 最新情绪：[当前读者情绪]
- 下阶段铺垫：[已为下一阶段埋好的引线]
```

## 四、上下文加载策略（写前读取）

### 写第1-10章时

```
读取：
  1. 02-大纲/卷纲.md（当前章节规划）
  2. 01-人物卡/（出场角色文件）
  3. 上一章全文（如有，用于衔接）
  4. 上一章记忆文件（如有）
```

### 写第11-20章时

```
读取：
  1. memory/phase-1-summary.md（第一阶段总结）
  2. 02-大纲/卷纲.md（当前章节规划）
  3. 01-人物卡/（出场角色文件）
  4. memory/ch-{上一章}-memory.md（上章记忆）
  5. memory/ch-{上上章}-memory.md（上上章记忆，可选）
  6. 上一章全文（仅用于衔接段落）
```

### 写第21章及以后（通用公式）

```
读取：
  1. memory/phase-{最新}-summary.md（最新阶段总结）
  2. memory/phase-{上一个}-summary.md（上一阶段总结，可选，仅跨阶段时加载）
  3. 02-大纲/卷纲.md（当前章节规划）
  4. 01-人物卡/（出场角色文件）
  5. memory/ch-{上一章}-memory.md
  6. memory/ch-{上上章}-memory.md（可选）
  7. 上一章全文（仅用于衔接）
```

### 关键原则

- **阶段总结**：始终只加载最新1份（跨阶段边界时加载最新2份）
- **章节记忆**：最多加载最近2份
- **全文回读**：只读上1章，且仅用于开头衔接段落的撰写
- **大纲和人物卡**：按需加载出场角色，不全量加载
- **伏笔查询**：需要确认伏笔状态时，读 `02-大纲/伏笔布局.md` 的追踪表

## 五、阶段总结的子Agent适配

### 并行模式下的上下文传递

当使用子Agent并行模式时，主Agent在派生子Agent时，除了传递大纲和人物卡，还需传递：

1. **最新阶段总结**（`memory/phase-{N}-summary.md`）
2. **批次起始前的章节记忆**（`memory/ch-{start-1}-memory.md` 和 `memory/ch-{start-2}-memory.md`）
3. **批次内的衔接说明**：主Agent在prompt中明确告知子Agent「你负责的第X章承接第X-1章的以下状态：[摘自记忆文件]」

### 子Agent内部的连续性

子Agent在批次内串行写作时，每章完成后仍需生成章节记忆文件。批次内后续章节通过读取本批次内的上一章记忆来维持连续性。

### 跨批次衔接

主Agent在所有子Agent完成后，执行一次全局衔接检查：
1. 读取每个批次边界的两份章节记忆
2. 检查场景/情绪/悬念是否连续
3. 如有断裂，派生修复任务

## 六、阶段总结触发脚本

在 `03-写作计划.json` 更新后，检查是否需要触发阶段总结：

```
IF completed_chapters % 10 == 0:
    读取本阶段(最近10章)的所有 ch-XXX-memory.md
    生成 memory/phase-{N}-summary.md
```

此逻辑内嵌在 Phase 3 的收尾步骤中，无需独立脚本。
